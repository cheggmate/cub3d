/src/parsing/
- parser.c: Orchestrates the reading of the .cub file.
- get_textures.c: Specifically handles the NO, SO, WE, EA lines.
- get_colors.c: Extracts and validates the F and C RGB values.
- get_map.c: Reads the map into a 2D array (char **map).
- validate_map.c: Contains the flood-fill logic to ensure the map is closed.

/src/raycasting/
- engine.c: The main game loop (mlx_loop_hook).
- dda.c: Implementation of the Digital Differential Analyzer algorithm.
- render.c: Logic for drawing vertical stripes on the screen.

/src/hooks/
- movement.c: Logic for updating player coordinates (W, A, S, D).
- rotation.c: Logic for updating the player's direction vector and camera plane.


To successfully validate your **cub3D** project, you must ensure your parser and engine handle a wide variety of misconfigurations. According to the project subject, any misconfiguration must result in the program exiting properly and returning "Error\n" followed by an explicit message.

### 1. File and Path Errors
* **Invalid Extension**: Testing with files that do not end in `.cub` (e.g., `map.txt` or `map.cub.rtf`).
* **Non-existent File**: Attempting to open a file that does not exist or for which you lack read permissions.
* **Missing Texture Files**: Providing a valid path in the `.cub` file, but the actual `.xpm` file is missing or corrupted.
* **Folder as Map**: Passing a directory path instead of a file path as the program argument.

### 2. Configuration Element Errors
* **Missing Elements**: A `.cub` file missing one or more required textures (NO, SO, WE, EA) or colors (F, C).
* **Duplicate Elements**: Providing the same identifier twice (e.g., two `NO` texture lines).
* **Invalid Identifiers**: Using incorrect type identifiers (e.g., `North` instead of `NO` or `Floor` instead of `F`).
* **Out-of-Order Map**: Placing map content before the texture or color definitions.
* **Incorrect RGB Values**: Providing colors outside the `` range, using non-numeric characters, or providing more/fewer than three values.

### 3. Map Content Errors
* **Invalid Characters**: Including characters in the map other than `0`, `1`, `N`, `S`, `E`, `W`, or spaces.
* **Multiple Spawns**: Providing more than one player start position (e.g., an `N` and an `E` on the same map).
* **Missing Spawn**: A map with no player start position at all
* **Open Map**: Any part of the map that is not completely surrounded by walls (`1`), including gaps that lead to the "outside" of the grid
* **Empty Lines in Map**: Including an empty line within the map content, which your parser must handle or reject based on whether it breaks the map structure.

### 4. Technical and Execution Errors
* **Memory Leaks**: All heap-allocated memory must be freed, especially when exiting due to one of the errors listed above.
* **Unexpected Quits**: The program must never experience a segmentation fault, bus error, or double free, even with the most "nasty" map mistakes.
* **Invalid Window Handling**: Changing window focus, minimizing, or resizing must not crash the program or break the rendering.
* **Unclean Exit**: Pressing `ESC` or clicking the red cross must close the window and free all resources cleanly.

### 5. Norm and Makefile Compliance
* **Norm Errors**: Any code, including bonus files, that does not comply with the school Norm will result in a 0.
* **Makefile Rules**: The Makefile must contain `$(NAME)`, `all`, `clean`, `fclean`, and `re`, and must not perform unnecessary relinking.
* **Compilation Flags**: Failure to use `-Wall -Wextra -Werror` during compilation.
